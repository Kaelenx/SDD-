# root_cause_analysis.md - 根因分析记录

## RCA-01：覆盖率工具不能依赖第三方包

| 项目 | 内容 |
|------|------|
| 现象 | 本地未确认安装 `coverage` 或 `pytest-cov`，如果项目依赖第三方覆盖率工具，评审环境可能无法直接运行 |
| 影响 | 单元测试覆盖率验收项可能无法稳定复现 |
| 直接原因 | 交付环境依赖状态不确定 |
| 根因 | 项目技术方案若绑定非必要外部依赖，会增加小型作业交付风险 |
| 修复方案 | 使用 Python 标准库 `trace` 编写 `tools/coverage_report.py`，不依赖第三方包 |
| 预防措施 | `plan.md` 中明确“标准库优先”，README 中提供标准运行命令 |
| 状态 | 已验证，`python tools/coverage_report.py` 可输出覆盖率 |

## RCA-02：开发期缺陷记录

### 项目形态不符合预期

| 项目 | 内容 |
|------|------|
| 现象 | 初始版本实现为 Python CLI 项目，用户进一步说明需要“前后端那种类型” |
| 影响 | 项目形态不符合提交预期，验收时可能被认为不贴合课程内容 |
| 直接原因 | 初始需求中未明确“前后端”形态，项目选型偏向小型命令行工具 |
| 根因 | 需求澄清阶段对“项目类型”这个业务边界确认不足 |
| 修复方案 | 保留学习任务追踪主题，重构为 REST 后端 + HTML/CSS/JS 前端 |
| 预防措施 | 在 `spec.md` 中明确项目形态、API 边界和前端交互边界 |
| 状态 | 已完成重构，并通过测试和浏览器验证 |

### 本机 8000 端口无法监听

| 项目 | 内容 |
|------|------|
| 现象 | 启动服务监听 8000 端口时出现 `PermissionError: [WinError 10013]` |
| 影响 | 按 8000 端口运行会失败 |
| 直接原因 | Windows 对该端口存在访问限制或保留规则 |
| 根因 | 端口选择未考虑本机系统保留端口范围 |
| 修复方案 | 将默认和文档示例端口改为 8765 |
| 预防措施 | README 中保留 `--port` 参数，评审环境如端口冲突可自行替换 |
| 状态 | 已修复并验证，`http://127.0.0.1:8765/api/health` 返回 ok |

### 覆盖率漏统计服务器线程

| 项目 | 内容 |
|------|------|
| 现象 | 初次改为前后端后，API 测试通过，但 `server.py` 覆盖率只有 29.1%，总覆盖率 68.0% |
| 影响 | 覆盖率结果低于真实测试覆盖情况 |
| 直接原因 | API 测试中的 HTTP 服务运行在子线程，覆盖率脚本只调用了 `sys.settrace` 跟踪主线程 |
| 根因 | 覆盖率工具设计时未考虑多线程后端测试场景 |
| 修复方案 | 在 `tools/coverage_report.py` 中增加 `threading.settrace(global_trace)` |
| 预防措施 | 后续如果增加后台线程或异步执行路径，覆盖率脚本需要同步覆盖执行上下文 |
| 状态 | 已修复，总覆盖率提升到 85.3% |

### 覆盖率脚本超时

| 项目 | 内容 |
|------|------|
| 现象 | 首次执行 `python tools/coverage_report.py` 已输出 90.9% 覆盖率和测试通过结果，但进程超过 30 秒被执行工具终止；功能升级后再次因每行路径解析开销超时 |
| 影响 | 覆盖率验收命令不够稳定 |
| 直接原因 | 初版使用 `trace.Trace` 对测试运行整体跟踪，后续版本在每行跟踪时重复做 Path.resolve |
| 根因 | 覆盖率统计范围和路径匹配策略未充分控制跟踪成本 |
| 修复方案 | 改为使用标准库 `sys.settrace`，只记录 `src/study_tracker` 下源码文件的行执行次数，并缓存规范化路径 |
| 预防措施 | 覆盖率脚本保持源码路径白名单，后续新增源码目录时同步更新 |
| 状态 | 已修复并复测，覆盖率命令约 3 秒完成 |

### 批量完成后选择状态未清空

| 项目 | 内容 |
|------|------|
| 现象 | 浏览器验证批量完成后，待完成筛选下任务列表为空，但选择栏仍显示“已选择 2 项” |
| 影响 | 前端状态与列表内容不一致，可能误导用户 |
| 直接原因 | `renderTasks` 在空列表分支提前返回，没有调用 `updateSelectionBar()` |
| 根因 | 新增批量选择功能后，空状态路径没有纳入状态同步验证 |
| 修复方案 | 在空列表渲染分支中也刷新选择栏状态 |
| 预防措施 | 浏览器验证覆盖“操作后列表为空”的边界状态 |
| 状态 | 已修复并验证，空列表下选择数为 0 且批量按钮禁用 |
